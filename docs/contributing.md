# Contributing

Contributions are welcome. Whether you want to report a bug, suggest an improvement, add a platform, or fix a typo - feel free to participate.

---

## Reporting an issue

Open an issue on the GitHub repository to:

- Report a bug or unexpected behavior.
- Suggest an improvement or new feature.
- Discuss an idea before submitting a pull request.

When reporting a bug, include:

- Your browser and version
- Your Node.js version (`node -v`) if applicable
- The framework or environment (React, Vue, vanilla HTML, etc.)
- Whether you are using the native Web Share API path or the fallback modal
- The error message and console output if applicable
- A minimal reproduction (CodePen, StackBlitz, or a minimal HTML file)

---

## Submitting a pull request

1. Fork the repository.
2. Create a dedicated branch:

```bash
git checkout -b my-proposal
```

3. Make your changes.
4. Run the test suite:

```bash
npm test
```

5. Build the package to verify output:

```bash
npm run build
```

6. Commit with a clear message:

```bash
git commit -m "Fix: description of the change"
```

7. Push the branch and open a pull request on GitHub.

---

## Running locally

```bash
# Install dependencies
npm install

# Run unit tests
npm test

# Run tests in watch mode
npm run test:watch

# Build all output formats (ESM, UMD, minified, .d.ts)
npm run build
```

## Best practices

- Stay true to the focused scope of the component: a share button with native API support and modal fallback.
- Only change what is necessary. Focused changes are easier to review.
- Add or update tests for any behavior you change. The test suite uses Vitest with happy-dom.
- Test in at least Chrome and Safari, the Web Share API behavior differs significantly between them.
- Verify that keyboard navigation (Tab, Escape, focus restoration) still works after any modal changes.
- Check that `no-shadow` mode continues to work after any template or lifecycle changes.
- When adding a new built-in language, add the full label set to `I18N_LABELS` in the component file.
- Consult the [changelog](../CHANGELOG.md) to understand the history of decisions.
